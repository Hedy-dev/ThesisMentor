# ThesisMentor

## Описание

ThesisMentor — система автоматизированной проверки текстов ВКР с генерацией рекомендаций по исправлению ошибок.

Текущий этап — реализация MVP (план минимум) в виде монолитного приложения с базовым ИИ-ядром.

Система решает задачи:
- структурирования текста ВКР;
- поиска типовых ошибок на основе правил и ML-моделей;
- генерации рекомендаций с использованием RAG;
- предоставления простого API для загрузки текста и получения результата.

---

## Установка

**Требования:** Python 3.10+

**1. Установка зависимостей**
Сначала установите все необходимые библиотеки:

```bash
pip install -r requirements.txt
```

**2. Настройка окружения**
Проект требует API-ключ GigaChat для глубокого анализа текстов.
Добавьте в файл `config.env` (создайте его в корне проекта) свои данные:

```env
GIGACHAT_CREDENTIALS=ваши_авторизационные_данные_из_личного_кабинета
```

**3. Запуск систем**
Запуск разделен на два этапа:

**Запуск Сервера (Backend):** Запустите `main.py`. При первом старте система скачает NLP-модели с Hugging Face и инициализирует векторную базу данных.

```bash
python main.py
```
*Дождитесь сообщения в консоли: «Все системы запущены».*

**Запуск Интерфейса (Frontend):**
После того как сервер успешно поднялся на порту 8000, запустите клиентскую часть:

```bash
python app.py
```

## Структура проекта

* `main.py` — точка входа FastAPI сервера.
* `core/parser.py` — логика извлечения структуры из `.docx`.
* `core/critic.py` — менеджер правил проверки.
* `core/generator_giga.py` — генерация советов на основе базы знаний (через GigaChat).
* `core/rules/` — папка с отдельными правилами (LLM-анализ, стилистика, структура).

> **Внимание:** Первый запуск может занять 10-20 минут в зависимости от скорости интернета, так как загружаются предобученные модели. Последующие запуски будут происходить быстрее.

## Архитектура решения

Система построена по модульному принципу в рамках монолитного сервиса. Запрос проходит линейно через три основных слоя: разбор документа, семантический анализ (поиск ошибок) и генерацию обратной связи.

```text
[ Клиент (app.py) ] ---> [ FastAPI (main.py) ]
                               |
                               v
[ 1. Парсер (.docx -> Граф) ] ---> [ 2. Критик (Локальные ML + Правила) ]
                                            |
                                            v
[ База Знаний (Qdrant) ] <---> [ 3. Генератор (RAG + GigaChat API) ]
                                            |
                                            v
                                  [ Возврат JSON-отчета ]
```

## Основные компоненты

### API (FastAPI)
- Загрузка текста (`/analyze`)
- Проверка состояния сервиса (`/health`)
- Получение результатов в формате JSON

### Парсер текста
- Разбор документа ВКР (`.docx`)
- Преобразование в древовидную структуру узлов
- Выделение секций, абзацев, заголовков с очисткой от технического мусора

### RAG модуль
- Индексация внешних документов (методические материалы по бизнес-проектам, методологии Бындю, Талера и др.)
- Векторизация с помощью `ruBERT-tiny2` и загрузка в Qdrant
- Поиск релевантного контекста для исправления найденных ошибок

### Модель "Критик"
- Набор правил для поиска ошибок (наследуются от `BaseRule`).
- Каждый тип ошибки — отдельный модуль.
- Использует локальные NLP-модели (например, `rubert-base-cased-nli-threeway`) для понимания контекста абзацев.
- Формирует:
  - описание ошибки
  - позицию в тексте (id узла)
  - статус (найдено/не найдено)

**Примеры типов ошибок:**
- Отсутствие обязательных глав (Введение, Заключение)
- Использование личных местоимений («я», «мой») в академическом тексте
- Отсутствие отражения практических результатов во введении (проверяется через LLM/NLI)

### Модель "Генератор"
- Получает описание ошибки и контекст из графа текста.
- Выполняет поиск по векторной БД (Qdrant) для получения методических рекомендаций.
- Обращается к GigaChat API для формирования связного и точного совета по исправлению.
- Обрабатывает кейс отсутствия релевантных данных, давая общие академические рекомендации.

Документ внутри системы представляется в виде ориентированного графа (дерева), состоящего из узлов (`ThesisNode`).

```json
{
  "root_id": "uuid-корневого-узла",
  "nodes": {
    "uuid-узла-1": {
      "id": "uuid-узла-1",
      "type": "SECTION",
      "title": "ВВЕДЕНИЕ",
      "level": 1,

      "children": ["uuid-абзаца-1", "uuid-абзаца-2"],
      "raw_text": null,

    "uuid-абзаца-1": {

      "clean_text": "очищенный текст в нижнем регистре"
    }
  }

```

## Поток обработки

1. Пользователь отправляет текст `.docx` через клиентское приложение.
2. Текст парсится в структуру словаря/графа (JSON-формат).
3. Модель "Критик" прогоняет граф через набор правил и ML-классификаторов, находя ошибки.
4. Ошибки собираются в единый список.
5. Модель "Генератор" получает список ошибок и соответствующие узлы текста.
6. Выполняется поиск по RAG (сбор подходящих методичек из базы знаний).
7. GigaChat генерирует конкретные рекомендации по исправлению каждой ошибки.
8. Сводный результат (список ошибок + советы) возвращается пользователю.

## Сравнение с аналогами

Традиционные системы проверки текстов (Антиплагиат, MS Word, Орфограммка) фокусируются на лексическом, грамматическом анализе или поиске прямых заимствований. Они не способны анализировать логику бизнес-проекта.

**ThesisMentor** отличается тем, что:
- **Анализирует смысл и структуру:** проверяет, соответствуют ли главы требованиям методических пособий (например, наличие конкретных метрик и гипотез в практической части).
- **Понимает контекст бизнес-дипломов:** использует заложенные методологии (карта гипотез, custdev) для оценки качества материала.
- **Предлагает решения, а не только указывает на ошибку:** благодаря интеграции RAG и LLM, студент получает конкретный совет (например: «Дополните введение результатами исследования CAC и LTV из Главы 3»), а не просто подсветку проблемной зоны.
